While writing some getXXX() and setXXX() methods in a web project, a question pops up: why I need to write these? After seconds of thinking, I cannot give myself a satisfying explanation, so I revisit my textbook and figure it out finally.

1. Why use private

Here is the data and methods visibility in Java

Modifier on members in a Class Access from the same class Access from the same package Access from a subclass in a different package Access from a different package
Public √ √ √ √
Protected √ √ √ -
Default √ √ - -
Private √ - - -

See an example

//Circle.java
public class Circle {
    double radius;
    static int numberOfObjects = 0;
    Circle() {
        radius = 1;
        numberOfObjects++;
    }
    Circle(double newRadius) {
        radius = newRadius;
        numberOfObjects++;
    }
    static int getNumberOfObjects() {
        return numberOfObjects;
    }
    double getArea() {
        return radius * radius * Math.PI;
    }
}

In this case, it’s more convenient to modify the radius and numberOfObjects, however, this is not a good practice:

  1. We use numberOfObjects to count how many Circle we have, but whoever use this class could arbitrarily change it. This is not our purpose.
  2. It’s difficult to maintain the code. Suppose we want to add some restrictions on the range of radius (for example, larger than 0), if we can directly access the radius field, it’s much more complicated.

To solve these drawback, it’s a good practice to declare data fields as private. A private field cannot be access by an object from outside the class that defines the private field.

2. get() and set()

Despite aforementioned drawbacks, we often need to retrieve and modify a data field. And this this why we need these two methods.

Let’s modify Circle class:

public class Circle {
    private double radius;
    private static int numberOfObjects = 0;
    public Circle() {
        radius = 1;
        numberOfObjects++;
    }
    public Circle(double newRadius) {
        radius = newRadius;
        numberOfObjects++;
    }
    public double getRadius() {
        return radius;
    }
    public void setRadius(double newRadius) {
        radius = (newRadius >= 0? newRadius : 0);
    }
    static int getNumberOfObjects() {
        return numberOfObjects;
    }
    double getArea() {
        return radius * radius * Math.PI;
    }
}

Now, because we only provide getNumberOfObject(), others cannot directly change its value, instead they need to create 10 Circle if they want numberOfObject to be 10. In line 16, we add a check easily to prevent some illegal values.

All in all, getRadius() and setRadius() like the only entry to the data fields (getRadius() to read and setRadius() to modify), and give us the total control over how the radius is accessed.

Other thoughts

  1. You can have only a getter. Then the property is read only. This should actually be the most common case.
  2. You can have only a setter, making the property configurable, but communicating that nothing else should depend on its value
  3. A setter can do sanity checks and throw IllegalArgumentException
  4. A setter can notify listeners of changes to the value
  5. You can have a setter that sets multiple fields together because they belong together conceptually.
